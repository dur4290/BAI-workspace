# /build — 계획하고 바로 만들기

아이디어를 작은 작업 카드로 정리한 뒤, 워크스페이스 루트의 `10-projects/작업이름/`에 첫 실행 가능한 결과물까지 만듭니다.

## 사용법

```text
/build 유튜브 자막을 공부 노트로 정리하기
```

인자 없이 실행하면 무엇을 만들고 싶은지 먼저 묻습니다.

## 원칙

- `/build`는 기본 진입점입니다. 계획과 구현을 사용자에게 두 번 시키지 않습니다.
- 아이디어가 막연하면 바로 코딩하지 말고 작업 카드를 먼저 만듭니다.
- 질문은 꼭 필요한 것만 최대 4개까지 묻습니다.
- 첫 결과물은 1시간 안에 확인 가능한 수준으로 줄입니다.
- 외부 API, 로그인 자동화, 배포, 과한 프레임워크를 기본안으로 잡지 않습니다.

## 실행 순서

### 1단계: 기준 읽기

먼저 `AGENTS.md`, `.codex/rules/build-workflow.md`, `.codex/references/work-readme-guide.md`, `.codex/references/work-quality-checklist.md`를 읽습니다.

현재 위치가 워크스페이스 루트인지 확인합니다. 루트는 `AGENTS.md`, `README.md`, `.codex/`, `10-projects/`가 함께 있는 폴더입니다. 현재 폴더가 `10-projects/` 안쪽이거나 다른 하위 폴더라면 새 `10-projects/`를 만들지 말고, 기존 루트의 `10-projects/`를 기준으로 작업합니다.

### 2단계: 작업 카드 만들기

요청을 아래 5줄로 정리합니다.

```text
목표:
입력:
출력:
성공 기준:
오늘 만들 최소 버전:
```

정보가 부족하면 한 번에 하나씩, 최대 4개까지만 묻습니다.

1. 어떤 파일, 폴더, 웹페이지, 메모를 입력으로 쓰나요?
2. 결과를 어떤 형태로 받고 싶나요? 예: Markdown, Python 스크립트, HTML, CSV/Excel
3. 성공했다고 보려면 무엇이 보여야 하나요?
4. 오늘은 어디까지 만들면 충분한가요?

### 3단계: 범위 줄이기

작업이 크면 로컬 파일 하나 또는 예시 데이터 하나로 실행되는 첫 결과물로 줄입니다. 출력은 Markdown·CSV·HTML·Python처럼 바로 열어볼 수 있어야 합니다.

로그인 자동화, 실제 결제·발송·삭제, 외부 API 키 발급, 웹 배포, 대형 프레임워크 도입은 나중으로 미룹니다.

### 4단계: 작업 폴더와 README 만들기

워크스페이스 루트의 `10-projects/작업이름/` 아래에 `README.md`, `src/`, `input/`, `output/`, `notes/`를 만듭니다.

`30-resources/examples/`는 참고 예시 전용입니다. 새 사용자 작업은 `30-resources/examples/` 안에 만들지 않습니다.

`README.md`에는 작업 카드, 만들 기능 체크리스트, 실행 방법, 확인 방법, 다음에 Codex에게 요청할 말을 포함합니다.

### 5단계: 첫 결과물 만들기

가능하면 빈 폴더와 README에서 끝내지 말고, 가장 작은 실행 결과물을 하나 만듭니다. 예: Python 스크립트 1개, 샘플 입력 파일 1개, 결과 Markdown/CSV/HTML 1개.

사용자가 "계획만" 원했거나 구현 정보가 부족하면 `/plan` 모드로 전환해 작업 카드만 저장합니다.

### 6단계: 점검과 보고

`.codex/references/work-quality-checklist.md`로 빠진 부분을 확인합니다.

마지막에는 아래만 짧게 알려줍니다.

```text
만든 위치:
첫 실행 방법:
확인 방법:
다음에 할 일:
```
