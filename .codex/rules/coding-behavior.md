# coding-behavior.md — 코딩 행동 기준

Karpathy의 LLM coding pitfalls 관찰과 커뮤니티 `andrej-karpathy-skills`의 4원칙을 Codex 작업용으로 정리한 문서입니다.

## 1. Think Before Coding

추측하지 말고, 혼란을 숨기지 말고, 트레이드오프를 드러냅니다.

- 구현 전에 가정과 성공 기준을 짧게 말합니다.
- 해석이 여러 개면 조용히 하나를 고르지 말고 선택지를 보여줍니다.
- 더 단순한 방법이 있으면 먼저 제안합니다.
- 불확실하면 멈추고 무엇이 헷갈리는지 말합니다.

## 2. Simplicity First

요청받은 문제를 푸는 최소 코드만 씁니다.

- 요청받지 않은 기능을 넣지 않습니다.
- 한 번만 쓰는 코드에 새 추상화를 만들지 않습니다.
- 필요하지 않은 설정 옵션, 확장성, 프레임워크를 추가하지 않습니다.
- 200줄로 쓴 코드가 50줄로 충분하면 줄입니다.

## 3. Surgical Changes

필요한 곳만 고칩니다.

- 주변 코드, 주석, 포맷을 임의로 개선하지 않습니다.
- 고장나지 않은 코드를 리팩터링하지 않습니다.
- 기존 스타일을 따릅니다.
- 관련 없는 dead code는 발견해도 삭제하지 말고 언급만 합니다.
- 내 변경 때문에 생긴 unused import, 변수, 함수는 정리합니다.

검사 질문: 바뀐 모든 줄이 사용자의 요청과 직접 연결되는가?

## 4. Goal-Driven Execution

작업을 검증 가능한 목표로 바꿉니다.

- "버그 고쳐줘" -> 재현 조건을 확인하고 고친 뒤 같은 조건으로 다시 확인
- "검증 추가해줘" -> 실패 입력을 먼저 정하고, 그 입력이 막히는지 확인
- "리팩터링해줘" -> 전후 동작이 같은지 확인

다단계 작업은 아래 형식으로 짧게 잡습니다.

```text
1. [작업] -> 확인: [검증 방법]
2. [작업] -> 확인: [검증 방법]
3. [작업] -> 확인: [검증 방법]
```

## 추가 기준

- 처음 보는 사람이 읽을 코드에는 짧은 설명을 남기되, 모든 줄에 억지 주석을 달지 않습니다.
- 처음 버전은 실행 가능한 작은 결과물을 우선합니다.
- 오류 메시지를 숨기지 말고, 원인과 다음 행동을 한국어로 설명합니다.
- 테스트 도구가 없으면 최소한 샘플 입력으로 실행 확인을 합니다.
